git 分支有幾種 branch, 長期分支: master branch, develop branch, 短期分支: feature branch, hotfix branch, release branch, 長期分支永遠不會被刪除, 短期分支都是被開發完就會 merge 到 develop 和 master

git flow 優缺點翻譯

  • 翻譯來源 - 出自Gitlab (https://about.gitlab.com/blog/2014/09/29/gitlab-flow/)
    Git flow 是 git 最早以前建議的工作流程, Git flow 主張專案中會有兩個長期分支:
  • 主分支 Master(master brnach)
  • 開發分支 Develop(develop branch)
    還有其他短期分支:
  • 功能分支 feature branch
  • 修復分支 hotfix branch
  • 預發分支 release branch
    Git flow 是一套定義明確的標準工作流程, 但是因為其相對複雜的關係帶來兩個問題:
  1. 開發人員必須轉換到 develop branch , master branch 上的程式碼必須保持為可以發布產品的狀態, 大部分的 branch 都由 master branch 分出來並且 merge 回 master branch, 由於大多數的工具都會選擇 master branch 為 default branch, 對於要一開始就切換到 develop 覺得很麻煩 (我: ????)
  2. 第二個問題是 hotfix 和 release branch 太複雜, 而且需要頻繁切換 branch, 在現在大多數專案都持續交付的情況下, 通常有 Bug 就在 develope 修復後 merge 到 master 中並且發布下個版本, 如果在 git flow 下, 開發人員常犯一個錯誤, 程式的修正僅 merge 到 master 中, 而忘記 merge 到 develop, 這些錯誤的根本原因是 git flow 太複雜了, 而且新版本的發布不代表要修復 BUG
